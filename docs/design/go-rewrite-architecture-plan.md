# Go 重构前置设计任务清单

## 使用方式

这份文档不再强调背景说明，而是作为团队协作的核心任务清单使用。目标是把 Go 重构前置设计拆成可并行、可评审、可验证的问题集合。

每个任务都包含：
- 任务目标
- 主要产出
- 需要回答的关键问题
- 验证任务是否已经清晰的问题
- 依赖关系
- 建议参与角色

建议所有专题负责人统一按以下结构交付：
1. 当前问题
2. 目标边界
3. 模块职责
4. 依赖关系
5. 关键决策点
6. 主要风险
7. 验证方法
8. 未决问题

---

## 任务 1：现状契约与不变量盘点

### 任务目标
梳理当前 Bella OpenAPI 的外部契约、内部关键行为和不可破坏语义，形成后续所有设计的统一输入。

### 主要产出
- API / DTO / streaming 契约清单
- 当前系统不变量清单
- 回归验证基线清单

### 需要回答的关键问题
- 当前对外暴露了哪些 API、请求结构、响应结构、错误结构
- 当前 streaming 的行为模型是什么
- 成功、失败、中断、超时、上游异常分别如何表现
- 哪些是外部依赖的正式契约，哪些只是当前实现细节
- 哪些语义在 Go 重构中绝对不能变化

### 验证任务是否清晰的问题
- 我们是否已经能列出“必须兼容”的 endpoint 清单？
- 我们是否已经能说明每类接口的输入、输出、错误、streaming 语义？
- 我们是否已经能明确哪些行为是“必须保持”的不变量？
- 如果让一个新同学阅读结果，是否能知道 Go 版本必须兼容什么？

### 依赖关系
- 无，最先开始

### 建议参与角色
- 熟悉 `endpoints/`、`protocol/`、OpenAI 兼容面的同学

## 任务 2：统一模块图与固定流程骨架设计

### 任务目标
给 Bella 定义统一的大模块视图、在线固定主流程、离线 Worker 固定主流程，以及模块依赖方向。

### 主要产出
- 一级模块图
- 在线固定骨架流程图
- 离线 Worker 固定骨架流程图
- 模块依赖约束与禁止依赖清单

### 需要回答的关键问题
- 在线链路的大模块有哪些
- 离线 Worker 链路的大模块有哪些
- 哪些模块属于前置校验、执行决策、调用执行、结果收口、全链路环绕
- 模块之间允许怎样依赖，哪些边界不允许穿透
- 在线链路和离线链路分别在哪里结束、在哪里开始

### 验证任务是否清晰的问题
- 我们是否已经能用一张图解释在线链路与离线链路？
- 我们是否已经能说明每个一级模块的职责一句话定义？
- 我们是否已经能明确“协议转换不能依赖哪些模块”这类边界约束？
- 如果不同专题负责人各自设计，是否能基于这张图避免互相冲突？

### 依赖关系
- 建议与任务 1 并行启动，但最终需要引用任务 1 的不变量

### 建议参与角色
- 总体架构 owner
- 各专题设计 owner 共同评审

## 任务 3：协议转换层与上游调用层边界设计

### 任务目标
把现有 adaptor 区域拆成“无状态协议转换层”和“上游调用层”，消除职责混杂。

### 主要产出
- Stateless Protocol Translation 设计
- Provider Invocation / Transport 设计
- adapter contract
- provider client contract
- streaming / error normalize 设计

### 需要回答的关键问题
- 哪些逻辑属于协议转换，哪些属于真正的上游调用
- adapter 的输入输出标准模型是什么
- streaming event 应该如何抽象
- provider error 如何统一归一
- adapter 如何做到无状态、纯转换、可单测
- online / offline 路径如何共享调用执行核心

### 验证任务是否清晰的问题
- 我们是否已经能明确说出 adaptor 不该做什么？
- 我们是否已经能定义 adapter 的最小职责边界？
- adapter 是否可以脱离 Redis、DB、日志系统独立测试？
- provider 调用失败时，错误归一化是否有明确归属？
- streaming 的起点、过程、终点是否有统一模型？

### 依赖关系
- 依赖任务 2 的统一模块边界
- 参考任务 1 的契约盘点结果

### 建议参与角色
- 熟悉 `protocol/`、streaming、厂商协议差异的同学

---

## 任务 4：路由与策略设计

### 任务目标
把 routing / policy 从“查 channel”升级为独立的执行决策模块。

### 主要产出
- routing input / output 模型
- route policy 设计
- channel capability 设计
- failover / fallback / direct mode / sticky routing 规则

### 需要回答的关键问题
- 路由输入有哪些
- 路由输出有哪些
- endpoint、model、channel 三者的核心关系是什么
- visibility、dataDestination、trial/free-tier、availability 如何参与路由
- 路由只做选择，还是也承担执行模式决策
- direct path / online path / offline path 的决策是在路由层还是编排层完成

### 验证任务是否清晰的问题
- 我们是否已经能写出“路由层的输入/输出”？
- 我们是否已经能区分“路由规则”和“执行逻辑”？
- 我们是否已经明确路由依赖 Registry，但不依赖 adaptor 细节？
- 当一个请求进入时，是否能解释它为什么选中某个 channel？
- failover 和 fallback 是否已经有清晰触发条件？

### 依赖关系
- 依赖任务 2
- 依赖任务 8 的部分领域模型定义

### 建议参与角色
- 熟悉 `ChannelRouter`、`ChannelService`、`ModelService` 的同学

---

## 任务 5：鉴权 / 治理 / 配额 / 安全设计

### 任务目标
把当前 interceptor 中混合的治理语义整理成清晰模块。

### 主要产出
- Auth / Governance 模块设计
- Quota / Rate Limit 模块设计
- Safety / Audit 模块设计
- 前置校验规则矩阵

### 需要回答的关键问题
- API Key、owner、org、role、certify、safetyLevel 的语义分别是什么
- 前置校验包含哪些规则
- quota、qps、monthQuota、concurrent limit 分别在哪里生效
- Safety 是准入门禁、执行中检查还是结果审计，或者三者兼有
- 这些治理能力在在线链路和离线链路中分别怎样生效

### 验证任务是否清晰的问题
- 我们是否已经能说清楚“前置校验”具体由哪些模块组成？
- 每类限制规则是否已经有明确归属？
- 同一个请求如果失败，哪些治理状态需要更新、哪些不需要？
- 离线任务是否也要复用相同治理规则？边界是否清楚？
- 是否已经能区分 auth / governance / quota / safety 的边界？

### 依赖关系
- 依赖任务 2
- 参考任务 1、任务 8

### 建议参与角色
- 熟悉 `ApikeyService`、`intercept/`、权限治理和租户语义的同学

---

## 任务 6：用量 / 计费 / 日志 / 观测统一收口设计

### 任务目标
解决 usage、billing、logging、metrics 长期彼此牵连的问题，定义统一收口模型。

### 主要产出
- usage extraction contract
- cost / billing boundary
- observability 设计
- terminal state consistency 规则
- 请求 / 响应 / 失败 / 审计日志归集方案

### 需要回答的关键问题
- usage 从哪里提取
- cost 在哪一层计算
- 成功、失败、超时、中断、流式异常终止时如何统一收口
- request/response log、audit log、metrics、trace 的职责边界是什么
- 哪些是执行核心必须提供的标准结果
- 失败请求是否也要计费、记录日志、记录审计

### 验证任务是否清晰的问题
- 我们是否已经能解释“失败场景也如何统一收口”？
- 是否已经明确 usage 与 cost 不是 adaptor 私有逻辑？
- 日志是否覆盖请求开始、执行中、结束、失败四类时点？
- 流式返回如果没有正常终止事件，是否仍然能统一结算？
- metrics、trace、audit 与普通 request log 的边界是否已清楚？

### 依赖关系
- 依赖任务 2、任务 3
- 参考任务 1

### 建议参与角色
- 熟悉 cost、日志、metrics、流式问题的同学

---

## 任务 7：离线 Worker / Queue / Async Execution 设计

### 任务目标
把离线 Worker 作为独立能力点设计，而不是在线链路的附属分支。

### 主要产出
- 离线 Worker 架构图
- Queue / Job lifecycle 设计
- Retry / Backoff / Timeout / Shutdown 设计
- Async execution observability 设计

### 需要回答的关键问题
- 任务如何进入系统
- 任务生命周期有哪些状态
- 调度、消费、执行、重试、取消、超时、关停如何定义
- 队列、任务状态、执行器、结果处理如何拆分
- 离线 Worker 与在线链路共享哪些能力，不共享哪些能力
- 离线链路的日志、指标、失败轨迹如何观测

### 验证任务是否清晰的问题
- 我们是否已经能画出完整离线链路而不是只说“有个 worker”？
- 是否已经能定义任务状态机？
- retry、backoff、timeout、shutdown 是否有明确责任归属？
- 离线任务执行完成后，usage、cost、result 由谁统一写回？
- 是否已经明确离线 Worker 不是在线模式的变体？

### 依赖关系
- 依赖任务 2
- 参考任务 4、任务 6、任务 8

### 建议参与角色
- 熟悉 `worker/`、`queue/`、`executor/` 的同学

## 任务 8：领域模型与持久化设计

### 任务目标
把 Java 现有 schema / jOOQ / POJO 世界映射成 Go 版本的领域模型与存储边界。

### 主要产出
- 领域模型图
- repository / persistence 分层设计
- JSON blob 治理方案
- schema 风险清单

### 需要回答的关键问题
- Endpoint、Model、Channel、Apikey、Space 等是否继续是一等对象
- 各对象之间的关系如何保留
- 哪些字段是业务语义核心，哪些是实现细节
- `channelInfo`、`priceInfo` 等 JSON 字段如何处理
- domain object、repository object、storage object 是否分层

### 验证任务是否清晰的问题
- 我们是否已经能列出 Go 领域模型中的一等实体？
- 哪些 Java 当前表结构语义必须保留，是否已经明确？
- 哪些 JSON blob 需要提升为显式结构，是否已有判断标准？
- repository 边界是否足够清楚，不会把业务语义压回存储层？
- 是否已经明确不能直接照搬 `new-api` 的扁平模型？

### 依赖关系
- 依赖任务 1、任务 2
- 会反向影响任务 4、任务 5

### 建议参与角色
- 熟悉 SQL、jOOQ、元数据、控制台数据结构的同学

## 任务 9：双栈迁移 / 验证 / 回滚设计

### 任务目标
给 Go 重构设计出可执行的双栈演进路径，而不是只停留在目标架构层。

### 主要产出
- 双栈迁移路线图
- 验证矩阵
- 回滚与上线门禁策略
- PoC 顺序建议

### 需要回答的关键问题
- 先做哪个能力 PoC
- 先切哪个 endpoint、provider、tenant、execution path
- shadow、replay、dual-run、rollback 怎么设计
- 如何验证 direct、streaming、offline worker 的一致性
- 如何定义“可以切流”的验收门槛

### 验证任务是否清晰的问题
- 我们是否已经能说出第一阶段 PoC 做什么、不做什么？
- 我们是否已经定义哪些验证通过后才允许下一阶段切流？
- 回滚条件是否清晰且可操作？
- 在线链路和离线链路是否分别有验证方案？
- 如果 Go 版本只做部分能力，是否也有明确的双栈共存策略？

### 依赖关系
- 强依赖任务 2～8 的设计结果

### 建议参与角色
- 架构 owner
- 熟悉发布、回滚、流量治理、线上稳定性的同学

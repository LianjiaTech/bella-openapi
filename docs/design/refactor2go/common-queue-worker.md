# Task 1 / Queue-Worker 执行语义

## 1. 范围说明

本文从横向公共语义中拆出 Queue / Worker 执行语义，聚焦跨能力点共享的排队执行、进度输出、异常映射与收口约束。

主索引文档：[`docs/design/refactor2go/task1-common-semantics.md`](./task1-common-semantics.md)

## 2. 不能改变的行为

- **不能改变的行为**
  - 请求一旦进入 queue / worker 执行，对外仍然继续表现为原有 endpoint 的协议语义，而不是暴露一套新的 worker 协议。
  - queue / worker 路径继续受渠道队列配置约束；因此 `queueName`、`queueMode` 不只是元数据字段，它们会直接决定请求是同步直连还是排队执行。
  - worker 执行链路继续要求“任务身份”和“原始请求身份”可稳定关联；排队、执行、流式进度、最终完成这些阶段之间不能失去同一笔请求的关联关系。
  - worker 路径下的非流式任务，继续表现为“任务完成时一次性返回最终结果”；流式任务继续表现为“先连续输出 progress，再单独给出完成态”。
  - worker 流式任务的 progress 顺序继续保持稳定，不能把中间进度和最终完成态混成单次返回，也不能在完成前丢失已经发出的进度事件。
  - worker 路径下的异常继续要映射回原 endpoint 的错误语义，例如最终状态码、错误消息、流式失败方式都要与该 endpoint 的主协议保持一致，而不是暴露 worker 内部错误模型。
  - worker 侧执行完成后，后续的用量、计费、限流释放、日志收口仍然必须发生；不能因为调用是异步任务就绕过治理与观测链路。

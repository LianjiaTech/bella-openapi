# Task 1 / 用量与计费语义

## 1. 范围说明

本文从横向公共语义中拆出用量与计费规则，聚焦跨能力点共享的 usage、priceInfo、成本归集与额度累计口径。

主索引文档：[`docs/design/refactor2go/task1-common-semantics.md`](./task1-common-semantics.md)

## 2. 不能改变的行为

- **不能改变的行为**
  - `usage` 继续是成本、指标和运营观测的基础输入；上游显式返回时优先使用上游值，未返回时允许按既有规则补算，但不能缺失后直接留空。
  - 成本继续以“命中的 endpoint + 该渠道的 `priceInfo` + 最终归一后的 `usage`”为核心输入；三者缺一都会改变计费结果，因此都必须在收口时被保留下来。
  - `priceInfo` 继续来自最终命中的渠道，而不是调用方请求参数；因此只要路由结果变化，计费结果就可能变化，这也是路由结果本身属于正式契约的原因之一。
  - 不同能力点继续允许把各自协议特有的 usage 归一成计费单位后再结算，例如 token、时长、次数、图片张数、网页抓取统计等；Go 重构后可以换实现，但不能改变这些 endpoint 级 usage 到计费单位的映射结果。
  - batch 请求继续允许在基础成本之上应用 `priceInfo` 中的 `batchDiscount`；也就是说“先按正常 usage 计算，再按批量折扣修正”这一外部计费语义不能变。
  - private 请求当前虽然继续生成 cost / costDetails，但默认不再把成本累计到共享额度计数器；这一“有成本明细，但不进入公共额度累计”的口径不能被误改成普通流量口径。
  - 非 private 请求在存在有效成本时，继续同时累计到当前 ak 和父 ak；因此父子 AK 的额度归集关系属于计费契约，而不是实现细节。
  - 当计费输入不足时，当前行为更接近“无法完成成本归集，而不是猜测一个费用值”；因此重构时不能在缺少 `priceInfo` 或 `usage` 的情况下自行编造计费结果。

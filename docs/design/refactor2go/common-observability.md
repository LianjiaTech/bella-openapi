# Task 1 / 观测与日志收口

## 1. 范围说明

本文从横向公共语义中拆出观测、日志与收口规则，聚焦跨能力点共享的完成时机、关联键和日志输出约束。

主索引文档：[`docs/design/refactor2go/task1-common-semantics.md`](./task1-common-semantics.md)

## 2. 不能改变的行为

- **不能改变的行为**
  - 无论同步还是流式，请求只有在“最终响应形态、错误结果、用量信息”基本确定后才算真正完成观测收口；不能把收口前移到业务执行前。
  - 流式请求也必须有完整收口动作，不能因为连接已经结束就跳过后续的用量、指标、成本、并发释放等处理。
  - `chat/completions` / `messages` / `responses` 三类流式能力继续各自以本协议的完成时机为准做收口，不能混成统一的粗粒度“连接断开即完成”语义。
  - `responses` 这类存在上游 `response_id` 的协议，继续要把该标识保留下来作为渠道侧关联键，不能在收口时丢失。
  - `requestId` 继续是单次请求的主关联键，贯穿错误、日志、限流释放、metrics 和成本收口；`traceId` 继续承担跨服务链路追踪职责，二者不能合并成一个字段。
  - mock 请求继续不进入正式日志/成本/治理口径；private 请求虽然继续保留日志与成本，但默认不进入常规并发治理口径。
  - Console 日志继续保持“两份不同用途的日志输出”这一行为：一份保留完整内容，另一份面向常规检索与展示。
  - 常规日志在超过 `bella.log.max-size-bytes` 时，继续允许裁剪大字段后输出；但完整日志继续不受这层裁剪影响。

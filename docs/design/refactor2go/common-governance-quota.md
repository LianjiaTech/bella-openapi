# Task 1 / 治理与配额语义

## 1. 范围说明

本文从横向公共语义中拆出治理、限流与配额规则，聚焦跨能力点共享的拦截顺序、并发计数和试用口径。

主索引文档：[`docs/design/refactor2go/task1-common-semantics.md`](./task1-common-semantics.md)

## 2. 不能改变的行为

- **拦截顺序与触发时机**
  - runtime 基础能力点继续在鉴权之后进入治理链路，当前顺序是：`Authorization` 之后先做 `QPS` 检查，再做 `MonthQuota` 检查，最后才进入具体业务逻辑。
  - 因此 QPS 与月额度都属于“请求进入 controller 前拦截”的规则；命中后继续直接失败，不会产生下游调用。
- **QPS 限流**
  - QPS 限流继续按 ak 维度生效，命中后返回 `429`，并继续附带 `Retry-After: 1` 响应头。
  - 这是入口级瞬时限流，与后面的 trial RPM / concurrent 限制不是一回事，二者不能混为一个规则。
- **月额度检查**
  - 月额度继续在业务执行前检查，按“本 ak 自身额度 + 父 ak 总额度”两层约束生效。
  - 当子 ak 未单独配置额度时，当前继续主要受父 ak 总额度约束；当子 ak 配了自身额度时，两层额度继续都可能拦截请求。
  - 月额度命中后继续返回 `429`，而不是业务成功后异步扣费失败。
- **试用额度与试用并发**
  - trial fallback 场景下，继续额外检查 free-tier 的每分钟请求数与并发数；任一超限都继续返回 `429`。
  - 这组规则只在“最低 safety level 且退化到试用渠道”时生效，不能扩大到普通正式渠道。
- **并发计数边界**
  - 并发计数不是统一在入口层自增，而是由各能力点在确认命中渠道之后显式增加；因此只有真正进入业务执行的请求才会计入并发。
  - 当前并发计数继续以 `akCode + entityCode(model/endpoint)` 为维度，而不是以 channelCode 为维度。
  - 请求结束后的并发释放继续依赖日志收口阶段统一 `decrement`；因此“有开始就要有收口”本身也是一个不能破坏的治理约束。
- **只对非 private 请求生效的规则**
  - 多个能力点当前继续只在 `!processData.isPrivate()` 时增加并发计数；private 请求默认跳过这层并发治理。
  - mock 请求继续不进入正式日志与成本收口，因此也不会沿用常规生产流量的治理统计口径。
